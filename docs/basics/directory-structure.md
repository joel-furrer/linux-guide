# Directory Structure

## Overview

Linux and most Unix-like systems organize files according to a well-defined directory structure. Rather than placing files arbitrarily, the system follows a convention known as the Filesystem Hierarchy Standard (FHS), which defines the purpose of major top-level directories. Understanding this structure makes it much easier to locate configuration files, programs, and data on any Linux system.

## Why It Matters

Knowing the standard directory structure allows you to:

- Quickly locate configuration files, logs, and installed programs without guessing.
- Understand documentation, tutorials, and software installation instructions that reference standard paths.
- Diagnose problems more effectively, since many issues relate to files being in unexpected or incorrect locations.
- Write scripts and automation that behave consistently across different Linux distributions, since most of them follow the same general structure.

## Concept

### The Root Directory

Every file and directory on a Linux system exists under a single top-level directory, represented by a forward slash (`/`), known as the root directory. Unlike some other operating systems, there are no separate drive letters; even external drives and other file systems are mounted as directories somewhere underneath the root directory.

### Key Top-Level Directories

The table below summarizes the purpose of the most important directories defined by the Filesystem Hierarchy Standard.

| Directory | Purpose |
|---|---|
| `/bin` | Essential user command binaries needed for basic system functionality, available to all users |
| `/sbin` | Essential system binaries, typically used for system administration tasks |
| `/etc` | System-wide configuration files |
| `/home` | Personal directories for regular users, typically one subdirectory per user |
| `/root` | The home directory of the root user |
| `/var` | Variable data that changes frequently, such as logs, caches, and spool files |
| `/tmp` | Temporary files, often cleared automatically on reboot |
| `/usr` | Secondary hierarchy containing the majority of user-facing programs, libraries, and documentation |
| `/opt` | Optional or third-party software packages, often self-contained |
| `/dev` | Device files representing hardware and virtual devices |
| `/proc` | A virtual directory exposing information about running processes and kernel parameters |
| `/lib` | Essential shared libraries needed by binaries in `/bin` and `/sbin` |
| `/mnt` | Conventional mount point for temporarily mounted file systems |
| `/media` | Conventional mount point for removable media, such as USB drives or optical discs |
| `/srv` | Data for services provided by the system, such as web or FTP server content |

### /etc in Detail

The `/etc` directory holds configuration files that control the behavior of the system and the software installed on it. Configuration files here are typically plain text and can usually be edited directly, although doing so generally requires root privileges.

### /var in Detail

The `/var` directory stores data that is expected to grow or change while the system is running. Common subdirectories include:

- `/var/log`: System and application log files.
- `/var/cache`: Cached data generated by applications.
- `/var/spool`: Data queued for later processing, such as print jobs or outgoing mail.

### /home in Detail

The `/home` directory contains a personal directory for each regular user on the system. By convention, each user's directory is named after their username, such as `/home/alice`. This is where personal files, downloads, and user-specific configuration are typically stored.

:::note

The root user's home directory is an exception to this pattern. Rather than being located under `/home`, it is found directly at `/root`.

:::

### /usr in Detail

Despite its name, `/usr` does not stand for "user." It is a secondary hierarchy that mirrors some of the structure found at the root level, containing subdirectories such as `/usr/bin`, `/usr/sbin`, and `/usr/lib`. The vast majority of installed software on a typical system resides somewhere under `/usr`.

### /proc in Detail

The `/proc` directory is a virtual file system; it does not contain real files stored on disk, but instead exposes live information about the kernel and running processes. Each running process is represented by a numbered subdirectory matching its PID, containing files that describe its current state.

## Examples

### Exploring the Directory Structure

List the contents of the root directory:

```bash
ls -l /
```

Display the structure of a directory and its subdirectories:

```bash
tree /etc -L 1
```

### Locating Configuration Files

View a system-wide configuration file, such as the hostname configuration:

```bash
cat /etc/hostname
```

List configuration files related to a specific service:

```bash
ls /etc/ssh
```

### Inspecting Logs

View the most recent lines of a system log file:

```bash
tail /var/log/syslog
```

:::note

Some distributions use `journalctl` to access logs managed by the system logging service, rather than relying solely on plain text files in `/var/log`.

:::

### Exploring /proc

View information about the current shell process using its PID:

```bash
echo $$
cat /proc/$$/status
```

List CPU information exposed by the kernel:

```bash
cat /proc/cpuinfo
```

### Checking Mounted File Systems

Display all currently mounted file systems and their mount points:

```bash
mount
```

Display disk usage for each mounted file system:

```bash
df -h
```

## Common Pitfalls

- **Storing personal files outside of /home**: Placing user data in unconventional locations makes backups, permissions, and multi-user management more difficult.
- **Editing files in /proc expecting persistence**: Since `/proc` is a virtual file system reflecting live kernel state, changes made there do not persist across a reboot in the same way as files stored on disk.
- **Assuming all distributions are identical**: While most major Linux distributions follow the Filesystem Hierarchy Standard closely, minor variations exist, particularly around `/usr` merging or additional vendor-specific directories.
- **Confusing /bin and /usr/bin**: On many modern systems, `/bin` is actually a symbolic link to `/usr/bin`, so distinguishing between them is largely historical rather than functional.
